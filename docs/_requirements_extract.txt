Требования к модулю ресурсно-сервисной модели
Термины и сокращения
Термин
Определение
РСМ
Ресурсно-сервисная модель: описание элементов ИТ-ландшафта и связей между ними.
Элемент РСМ
Объект модели: бизнес-сервис, приложение, сервер, БД, очередь, контейнер, сетевой элемент и т.д.
Correlation engine
Внешняя система или модуль, который группирует события и формирует корреляционный контекст на основании правил, времени и данных РСМ.
2. Контекст и цели
2.1. Проблема или потребность
В системах мониторинга события часто приходят как отдельные технические алерты: hostname, IP, метрика, текст ошибки, имя сервиса. Без модели зависимостей сложно понять, какие алерты относятся к одному сервису, какой бизнес-сервис затронут и какие компоненты могут быть связаны между собой.
2.2. Цель
Цель модуля РСМ - обеспечить единое хранение ресурсно-сервисной модели и предоставлять внешним системам структурированные данные об объектах, связях и цепочках влияния, чтобы использовать РСМ как шаблон для корреляции событий и обогащения инцидентов.
2.3. Заинтересованные стороны
Product Owner и бизнес-заказчик — получают формализованные требования к модулю и границы MVP.
Системные аналитики и архитекторы — используют требования для проектирования модели данных, API и сценариев интеграции.
Команды эксплуатации и мониторинга — используют РСМ для понимания связей между техническими ресурсами и сервисами.
Внешние системы мониторинга и корреляции — используют API РСМ для сопоставления алертов с объектами и построения корреляционного контекста.
2.4. Допущения и ограничения
РСМ не выполняет мониторинг инфраструктуры и не заменяет систему мониторинга.
РСМ не хранит полный поток сырых алертов и метрик.
РСМ предоставляет модель, связи и контекст; финальное решение о группировке событий и первопричине принимает внешняя система корреляции.
Без актуальных данных и корректных связей качество корреляции будет ограничено качеством самой модели.
3. Общее описание решения
3.1. Решение включает
ведение справочника элементов ресурсно-сервисной модели;
ведение направленных связей между элементами;
импорт и экспорт элементов, связей и атрибутов;
API для поиска объектов, получения карточек, прямых связей, графа зависимостей и impact analysis;
сценарии использования РСМ как шаблона для связывания алертов во внешних системах;
аудит изменений модели и базовую ролевую модель.
3.2. Основной сценарий использования
Во внешнюю систему мониторинга поступает группа алертов за короткое временное окно.
Внешняя система передает идентификаторы алертов в API РСМ: hostname, IP, serviceCode, applicationCode, externalId.
РСМ сопоставляет идентификаторы с объектами модели.
Внешняя система получает граф зависимостей по найденным объектам.
Correlation engine определяет, какие алерты находятся в одной ресурсно-сервисной цепочке.
Связанные алерты объединяются в один инцидент или корреляционную группу.
Инцидент обогащается данными о бизнес-сервисе, критичности, окружении, владельцах и потенциальной зоне первопричины.
3.3. Логика использования РСМ как шаблона для связывания
РСМ используется как эталонная карта зависимостей. Если несколько алертов пришли в одном временном окне и их объекты находятся в одной цепочке РСМ, внешняя система может рассматривать их как связанные события. Например: база данных → приложение → бизнес-сервис. РСМ не заменяет correlation engine, но предоставляет ему структуру, по которой события можно связать между собой.
4. Состав и архитектура
4.1. Компоненты
Компонент
Назначение
Примечание
Хранилище РСМ
Хранит элементы модели, атрибуты, внешние идентификаторы и связи.
Основной источник данных для API.
Модуль управления элементами
Создание, просмотр, изменение и удаление элементов РСМ.

Модуль управления связями
Создание, изменение и проверка направленных связей между элементами.

Модуль импорта/экспорта
Загрузка и выгрузка элементов, связей и атрибутов.
Поддерживает стартовое наполнение и обмен данными.
API РСМ
Отдает данные внешним системам мониторинга и корреляции.
Ключевой интерфейс для использования РСМ как шаблона корреляции.
Аудит
Фиксирует изменения элементов, связей, импортов и экспортов.
Нужен для контроля качества модели.
4.2. Потоки и зависимости
Пользователь → UI РСМ → создание и сопровождение элементов и связей.
Файл / внешняя выгрузка → импорт РСМ → создание или обновление элементов и связей.
Внешняя система мониторинга → API РСМ → сопоставление алертов с объектами модели.
Correlation engine → API РСМ → получение графа зависимостей и контекста для связывания событий.
РСМ → экспорт → передача модели или её части во внешние системы.

6. Данные и учёт объектов
Основные сущности: элемент РСМ, связь РСМ, атрибут элемента, внешний идентификатор, пользователь, роль, запись аудита.
Элементы РСМ должны иметь внутренний идентификатор и набор внешних идентификаторов для сопоставления с внешними системами.
Связи должны быть направленными и должны хранить тип связи, источник и статус.
Удаление элементов рекомендуется выполнять через архивирование, чтобы не терять исторический контекст модели.
Импорт должен проверять обязательные поля, дубли, корректность типов и валидность связей.
7. Интерфейс и взаимодействие с пользователем
Список элементов РСМ с поиском и фильтрами.
Карточка элемента с атрибутами, внешними идентификаторами, владельцами и связями.
Экран или область визуального просмотра графа зависимостей.
Формы импорта и экспорта модели.
Журнал аудита изменений.
8. Интеграции и API
API РСМ должно отдавать не только карточку объекта, но и корреляционный контекст: найденный объект РСМ, тип объекта, критичность, окружение, владельцев, внешние идентификаторы, прямые связи, граф зависимостей, направление связей, типы связей, затронутые бизнес-сервисы и нижележащие компоненты.
API
Назначение
GET /api/v1/resources/{id}
Получение карточки объекта РСМ.
GET /api/v1/resources/search
Поиск объекта по hostname, IP, serviceCode, externalId и другим идентификаторам.
GET /api/v1/resources/{id}/relations
Получение прямых связей объекта.
GET /api/v1/resources/{id}/graph?depth=N
Получение графа зависимостей на заданную глубину.
GET /api/v1/resources/{id}/impact
Получение бизнес-сервисов, которые могут быть затронуты проблемой ресурса.
POST /api/v1/resources/resolve
Пакетное сопоставление группы алертов или идентификаторов с объектами РСМ.
POST /api/v1/correlation/context
Получение общего графа и корреляционного контекста по группе объектов.

Требование
Производительность
API поиска объекта и получения ближайших связей должен быть пригоден для использования во внешнем процессе корреляции событий.

API должно иметь OpenAPI/Swagger-спецификацию для подключения внешних систем.
10. Отчёты и выходные артефакты
Выгрузка всей модели или выбранной части модели в JSON.
Выгрузка элементов и связей в XLSX/CSV.
Отчёт о результатах импорта: создано, обновлено, пропущено, ошибки.
Журнал аудита изменений элементов, связей, импортов и экспортов.
11. Хранение и жизненный цикл данных
Модуль должен хранить активные и архивные элементы модели.
Модуль должен хранить активные и архивные связи между элементами.
Для элементов и связей должна быть доступна дата последнего изменения.
История изменений должна позволять восстановить, кто и когда изменил ключевые данные модели.
