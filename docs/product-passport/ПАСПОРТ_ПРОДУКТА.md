## 1. Маркетинговое название продукта

**OmniSight РСМ**  
*(OmniSight — модуль ресурсно-сервисной модели ИТ-ландшафта)*

---

## 2. Маркетинговое описание продукта

**OmniSight РСМ** — это платформа для учёта и визуализации ИТ-ландшафта: единый реестр конфигурационных единиц, направленных зависимостей между ними и интерактивная карта топологии. Продукт даёт командам эксплуатации и внешним системам мониторинга **эталонную карту зависимостей** — чтобы сопоставлять разрозненные технические алерты с объектами модели, понимать бизнес-контекст инцидента и ускорять поиск первопричины.

OmniSight **не заменяет** систему мониторинга и correlation engine, а выступает **источником правды** о том, как устроен ваш ИТ-ландшафт и как компоненты связаны между собой.

---

## 3. Суть продукта

РСМ (ресурсно-сервисная модель) — это структурированное описание элементов ИТ-инфраструктуры (серверы, БД, приложения, бизнес-сервисы и др.) и **направленных связей** между ними (`depends_on`, `hosted_on`, `part_of`, `uses` и др.).

**Ключевая идея:** когда в мониторинг приходит группа алертов (hostname, IP, serviceCode, externalId…), внешняя система корреляции обращается к API РСМ, получает сопоставленные объекты, граф зависимостей и обогащение (критичность, окружение, затронутые бизнес-сервисы, зону первопричины). На этой основе correlation engine принимает решение об объединении алертов в один инцидент.

**В одной фразе:** реестр ИТ-ландшафта с направленными связями и API для корреляции алертов.

## 6. Какие проблемы решает

1. **Разрозненные технические алерты** — hostname, IP, метрика, код сервиса приходят отдельно; без модели зависимостей непонятно, относятся ли они к одному инциденту.
2. **Отсутствие связи «техника → бизнес»** — сложно быстро ответить, какой бизнес-сервис пострадал при падении БД или сервера.
3. **Устаревшая или разрозненная CMDB** — данные об активах живут в Excel, в головах инженеров или в несвязанных системах.
4. **Долгий разбор инцидентов** — без карты зависимостей инженер вручную выясняет цепочку «БД → приложение → бизнес-сервис».
5. **Низкое качество корреляции** — correlation engine не может группировать события без эталонной модели связей.
6. **Нет аудита изменений модели** — непонятно, кто и когда изменил элемент или связь, что снижает доверие к данным.

## 8. Как работает продукт (техническая часть)

### Архитектура

```
┌─────────────────┐     JWT      ┌──────────────────┐     SQL      ┌─────────────┐
│  React UI       │─────────────▶│  FastAPI API     │─────────────▶│ PostgreSQL  │
│  (Vite, :5173)  │              │  (:8000)         │              │  (:5432)    │
└─────────────────┘              └────────┬─────────┘              └─────────────┘
                                          │
┌─────────────────┐     X-API-Key         │
│  Demo CE        │───────────────────────┘
│  (:8090)        │  POST /api/v1/correlation/ingest
└─────────────────┘
         ▲
         │ алерты (hostname, IP, serviceCode…)
┌────────┴────────┐
│ Zabbix,         │
│ Prometheus,     │
│ Nagios, APM     │
└─────────────────┘
```

### Стек технологий

| Слой | Технология |
|------|------------|
| Frontend | React 19, TypeScript, Vite, Tailwind CSS 4, TanStack Query, React Flow |
| Backend | Python 3.12+, FastAPI, SQLAlchemy, Alembic, Pydantic |
| БД | PostgreSQL 16+ |
| Деплой | Docker Compose (dev/prod), Nginx |

### Модель данных

- **11 типов CI:** Business Service, IT Service, Application, Technical Component, Server, Virtual Machine, Container, Database, Queue, Network Element, External Service.
- **7 типов связей:** `depends_on`, `affects`, `part_of`, `hosted_on`, `uses`, `linked_to`, `reserves`.
- **Внешние идентификаторы для корреляции:** hostname, IP, serviceCode, applicationCode, externalId, cmdbId.

### Основной API-сценарий корреляции (§3.2 ТЗ)

```
1. Мониторинг → группа алертов за временное окно
2. Correlation Engine → POST /api/v1/correlation/ingest
   Body: { "alerts": [{ "hostname": "app-01" }, { "ip": "10.0.0.5" }, …] }
3. РСМ → resolve (сопоставление с CI) + graph (граф зависимостей)
4. РСМ → correlation.chain_related (алерты на одной цепочке depends_on?)
5. РСМ → enrichment (имя, тип, критичность, окружение, бизнес-сервисы)
6. РСМ → potential_root_cause_zone (листья цепочки depends_on)
7. CE → решение об объединении в инцидент (вне РСМ)
```

### Ключевые модули backend

- `services/ci/` — CRUD элементов, import/export
- `services/relations/` — связи, валидация (циклы, архив, битые ссылки)
- `services/rsm/` — граф, topology, impact, search
- `services/rsm_correlation.py` — batch resolve и корреляционный контекст
- `services/autodiscover/` — pipeline: коннектор → discovery → mapping → apply
- `services/audit.py` — журнал изменений

### Безопасность и эксплуатация

- JWT для UI, `X-API-Key` для интеграций
- Health: `/health/live`, `/health/ready`
- Rate limiting, CORS, security headers
- OpenAPI: `http://localhost:8000/docs`

### Сценарий 1. Корреляция алертов

| | |
|---|---|
| **Цель** | Понять, относятся ли несколько технических алертов к одному инциденту, и получить бизнес-контекст для разбора |
| **Роль** | Инженер мониторинга (L2) или correlation engine |
| **Предусловия** | В РСМ заведена цепочка зависимостей (например PAY: `demo-db` → `demo-app` → `demo-biz`); у CI заполнены внешние ID (hostname, IP, serviceCode и др.) |
| **Триггер** | За короткое окно (5–10 мин) пришла группа алертов из разных систем мониторинга |

**Ход сценария**

1. **Входящие события.** Мониторинг фиксирует алерты с разными идентификаторами:
   - Zabbix: `hostname: app-01` (CPU > 90%)
   - Prometheus: `ip: 10.0.0.5` (мало места на диске)
   - Nagios: `externalId: ext-db-1` (ошибки подключения к БД)
   - APM: `serviceCode: PAY`, `applicationCode: PAY-APP` (рост латентности)
2. **Передача в РСМ.** Correlation engine отправляет batch в API:
   `POST /api/v1/correlation/ingest` с заголовком `X-API-Key`.
3. **Сопоставление (resolve).** РСМ находит объекты модели по hostname, IP, externalId, serviceCode и возвращает список resolved / unresolved.
4. **Построение контекста.** РСМ отдаёт:
   - граф зависимостей по найденным CI;
   - `chain_related: true/false` — все ли алерты на одной цепочке `depends_on`;
   - **обогащение** — тип, окружение, критичность, затронутые бизнес-сервисы;
   - **зону первопричины** (`potential_root_cause_zone`) — кандидаты в корень цепочки.
5. **Решение CE.** На основе ответа РСМ correlation engine решает: объединять алерты в один инцидент или оставить раздельно.  
   *В демо:* при `chain_related=true` и 4/4 resolved → инцидент «INC-DEMO: деградация стека PAY».
6. **Проверка в UI (опционально).** Тот же ingest можно повторить вручную на экране **Корреляция** (`/correlation`) для отладки правил.

**Ожидаемый результат**

- Вместо 4 несвязанных тикетов — **один инцидент** с гипотезой первопричины (например, `demo-db`).
- В карточке инцидента видны: затронутый бизнес-сервис, критичность, окружение.

**Ценность:** сокращение MTTR, меньше «шума» от дублирующих алертов, понятный бизнес-контекст без ручного поиска в CMDB.

**Скриншоты:** [`10-demo-ce-result.png`](./screenshots/10-demo-ce-result.png), [`06b-correlation-result.png`](./screenshots/06b-correlation-result.png)

### Сценарий 2. Impact-анализ на карте зависимостей

| | |
|---|---|
| **Цель** | До начала работ или при сбое понять, какие сервисы и компоненты окажутся под ударом |
| **Роль** | Инженер эксплуатации, дежурный смены |
| **Предусловия** | В РСМ построена модель с направленными связями; элементы имеют статус `active` |
| **Триггер** | Планируется обслуживание сервера / VM / БД **или** поступил алерт на инфраструктурный компонент |

**Ход сценария**

1. Пользователь открывает **Карту зависимостей** (`/graph`).
2. В поле **«Корневой элемент»** выбирает точку отсчёта — Business Service (`demo-hub-biz`) или компонент, который планируют трогать (`demo-hub-srv`).
3. Задаёт **глубину** обхода (например, 3) и при необходимости фильтр по типу связи.
4. На графе видит цветовую легенду:
   - **корень** (красный) — выбранный элемент;
   - **бизнес-путь** (зелёный) — цепочка до бизнес-сервиса;
   - **зона влияния** (жёлтый) — что пострадает при отказе корня;
   - **компоненты ниже** (синий) — инфраструктура «под» приложением.
5. В боковой панели читает сводку: бизнес-путь, затронутые Business Service, список компонентов ниже.
6. Принимает решение: окно работ, эскалация владельцу (`owner`), приоритет инцидента по **критичности** CI.

**Пример на демо-данных**

```
demo-hub-biz (Business Service)
    └── depends_on → demo-hub (Application)
            ├── depends_on  → demo-hub-db
            ├── hosted_on   → demo-hub-srv   ← плановые работы здесь
            ├── part_of     → demo-hub-vm
            └── uses / linked_to / affects / reserves → …
```

**Ожидаемый результат**

- Понятный **blast radius**: какой бизнес-сервис и какие приложения/БД затронуты.
- Обоснованное согласование работ с владельцем сервиса.

**Ценность:** снижение риска «случайно положить» критичный сервис; быстрый ответ на вопрос «что сломается, если упадёт X».

**Скриншот:** [`05b-graph-topology.png`](./screenshots/05b-graph-topology.png)

### Сценарий 3. Наполнение модели (Autodiscover и импорт)

| | |
|---|---|
| **Цель** | Быстро и достоверно наполнить или обновить реестр CI и связей без ручного ввода сотен записей |
| **Роль** | Администратор модели (роль editor / admin) |
| **Предусловия** | Пользователь авторизован с правами на создание CI; настроен коннектор или подготовлен файл импорта |
| **Триггер** | Появился новый контур (кластер, стенд, филиал) **или** нужна миграция данных из Excel / старой CMDB |

**Ход сценария — вариант A: Autodiscover**

1. **Настройка источника.** В **Настройки → Коннекторы** создаёт коннектор (SSH/host или file).
2. **Запуск обнаружения.** На экране **Элементы** (`/inventory`) нажимает **Autodiscover** — система собирает данные с указанных хостов или из файла.
3. **Черновой mapping.** РСМ предлагает auto-mapping: какие поля станут CI, hostname, IP, тип элемента.
4. **Подтверждение.** Администратор просматривает черновик, правит при необходимости и **подтверждает** создание/обновление CI.
5. **Связи.** При необходимости добавляет зависимости вручную или через импорт связей.

**Ход сценария — вариант B: импорт файла**

1. Готовит файл **JSON / CSV / XLSX** (или использует готовые демо-файлы из `fast_start/`).
2. На **Элементах** нажимает **Импорт**, загружает файл.
3. Получает **отчёт импорта**: создано / обновлено / ошибки.
4. При необходимости импортирует связи отдельным файлом.

**Завершение (общее для A и B)**

6. На **Обзоре** проверяет метрики: число элементов, связей, **целостность связей** (проблем нет / есть предупреждения).
7. На **Карте** запускает **«Проверить модель»** (валидация FR 19).
8. В **Аудите** (`/audit`) видит запись об импорте / autodiscover с diff old/new.

**Ожидаемый результат**

- Актуальная модель, готовая к **корреляции алертов** (resolve по hostname, IP, externalId).
- Прозрачная история изменений для аудита качества данных.

**Ценность:** быстрый старт без «пустой CMDB»; меньше ошибок ручного ввода; контроль качества через валидацию и аудит.

### Сценарий 4. Интеграция Correlation Engine через API

| | |
|---|---|
| **Цель** | Подключить существующий или новый correlation engine к РСМ без доработки ядра продукта |
| **Роль** | Разработчик интеграций, SRE, архитектор AIOps |
| **Предусловия** | РСМ развёрнут; модель наполнена; есть доступ к Swagger и API Key |
| **Триггер** | В организации принято решение использовать РСМ как **источник контекста** для группировки алертов |

**Ход сценария**

1. **Получение доступа.** В **Настройки → API** копирует `X-API-Key` (в dev: `omnisight-api-key-dev`).
2. **Изучение контракта.** Открывает Swagger (`http://localhost:8000/docs`), ключевые методы:
   - `POST /api/v1/correlation/ingest` — batch сопоставление + граф + корреляционный контекст;
   - resolve, graph, impact — при необходимости точечных запросов.
3. **Реализация вызова.** В CE при накоплении batch алертов за окно времени:
   ```http
   POST /api/v1/correlation/ingest
   X-API-Key: <ключ>
   Content-Type: application/json

   {
     "alerts": [
       { "hostname": "app-01" },
       { "ip": "10.0.0.5" },
       { "externalId": "ext-db-1" },
       { "serviceCode": "PAY", "applicationCode": "PAY-APP" }
     ],
     "source": "my-ce",
     "depth": 3
   }
   ```
4. **Использование ответа** в логике CE:
   - `resolve.resolved` / `resolve.unresolved` — что сопоставилось;
   - `correlation.chain_related` — можно ли считать алерты связанными;
   - `correlation.enrichment` — обогащение для тикета/инцидента;
   - `correlation.potential_root_cause_zone` — кандидаты первопричины.
5. **Правило группировки (пример).** CE создаёт один инцидент, если `chain_related=true` и resolved ≥ N; иначе — отдельные инциденты или эскалация оператору.
6. **Тест на стенде.** Перед prod прогоняет сценарий на **Demo CE** (`http://localhost:8090`) — кнопка «Отправить 4 алерта в РСМ» воспроизводит полный цикл.

**Ожидаемый результат**

- CE получает структурированный контекст из РСМ за один API-вызов.
- Интеграция документирована через OpenAPI; доработка ядра РСМ не требуется.

**Ценность:** ускорение внедрения AIOps; чёткое разделение ответственности — **РСМ даёт модель, CE принимает решение** о группировке.
