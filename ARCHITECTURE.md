# OmniSight Architecture

**English** · [Русская версия](ARCHITECTURE.ru.md)

## Layering

### Backend (`apps/api`)

```
api/v1  >  core/deps/  >  services/*  >  repositories/*  >  models/
```

| Layer | Responsibility | May import |
|-------|----------------|------------|
| `api/` | HTTP routing, auth, DTO binding | `services/`, `schemas/`, `core/deps` |
| `services/` | Domain logic, orchestration | `repositories/`, `models/`, `schemas/`, `core/` |
| `repositories/` | SQLAlchemy data access | `models/`, `core/` ? **not** `services/` |
| `core/` | Infra: auth, cache, DB, DI | `models/` sparingly |

**Transaction policy** (`core/async_unit_of_work.py`, `core/session_commit.py`):

- CRUD services commit after each successful mutation
- Import pipelines batch mutations; caller commits once via `async_transactional_write_session`
- Read-only paths never commit

**Database access:**

| Module | Use |
|--------|-----|
| `core/database_async.py` | All HTTP handlers, worker runtime, async cache API |
| `core/database.py` | Sync `engine` + `sync_session()` ? schema bootstrap, CLI scripts, legacy sync cache helpers |
| `core/migrations.py` + `alembic/` | Schema evolution (`alembic upgrade head`, brownfield `stamp head`) |
| `core/sync_transaction.py` | Legacy deferred-commit sync session (tests / bridged imports) |

### Frontend (`apps/web`)

```
features/*/XPage.tsx  >  hooks/useXPage.ts  >  shared/api/*  +  shared/components/*
```

| Layer | May import |
|-------|------------|
| `features/` | `shared/`, `lib/`, `context/` |
| `shared/` | `shared/`, `lib/` ? **not** `features/` |
| `shared/components/graph/` | Graph canvas, legend, embedded read-only panel |
| `components/` (shell) | `Layout`, `ProtectedRoute`, `PageLoader`, `ui.tsx` barrel ? **not** `features/` |
| `shared/components/` | Reusable UI: tables, forms, modals (`ui-primitives`, `EnvironmentInput`), graph canvas |

**Import rules:**

- App shell only from `@/components/*`
- Reusable widgets from `@/shared/components/*` or `@/components/ui` (barrel re-exports shared primitives)
- Do **not** import `@/components/Modal` ? use `@/components/ui`
- Use `@/` path alias (`@/shared/...`, `@/features/...`) for all imports

Styles live in `src/styles/` (`tokens.css`, `theme.css`, `base.css`, `components.css`, `graph.css`, `layout.css`), imported via `src/index.css`. See **`apps/web/docs/STYLING.md`** for when to use Tailwind vs CSS classes vs tokens.

**Shared client contract:** `packages/omnisight-contract` (`@omnisight/contract`) ? domain constants for web and future clients; generated by `sync_domain_constants.py`.

**Design tokens** (`tokens.css`): radii, spacing, caption size (`--rsm-font-caption`), semantic palette (`--rsm-color-*`). Theme-aware semantic text and alerts (`--text-warning`, `--alert-warning-*`, etc.) live in `theme.css`. UI classes: `.btn-*`, `.alert-*`, `.hint`, `.text-warning` in `components.css`.

Route pages use `React.lazy()` + `Suspense` in `App.tsx`. Heavy vendor chunks: `vite.config.ts` `manualChunks`.

Path alias: `@/` > `src/` (Vite + `tsconfig.app.json`) ? **use for all new and migrated imports**.

## Dependency injection (API)

Providers live in `core/deps/` (`read.py`, `write.py`, `domain.py`). Re-exported from `core.deps` for route imports.

Write path ? `core/deps/write.py` (`get_*_write_port`, transactional variants):

- `get_ci_write_port`, `get_relation_write_port`, `get_graph_layout_write_port`, `get_correlation_write_port`
- `get_user_write_port`, `get_ci_type_write_port`, `get_ci_import_export_write_port`, `get_autodiscover_write_port`
- `get_transactional_ci_import_export_write_port`, `get_transactional_autodiscover_write_port`
- Legacy aliases: `get_ci_service`, `get_relation_service`, ? (same providers)

Handlers call `await write_service.method(...)`. All HTTP mutations use the **async write pool**:

- **Native async ORM:** CI, relations, CI types, graph layout, users, correlation ingest
- **Bridged ``await session.run_sync``:** CI/relation import-export, autodiscover (connector CRUD, scan)
- **Transactional async write pool:** mapped import apply, autodiscover apply (`async_transactional_write_session`, deferred commit)

Read path ? `core/deps/read.py` (`get_*_read_port`, optional `DATABASE_READ_URL` replica):

- `get_ci_read_port`, `get_relation_read_port`, `get_topology_read_port`, `get_dashboard_read_port`
- `get_search_read_port`, `get_correlation_read_port`, `get_graph_layout_read_port`
- `get_audit_read_port`, `get_user_read_port`, `get_ci_type_read_port`
- `get_ci_import_export_read_port`, `get_relation_import_export_read_port`, `get_autodiscover_read_port`
- Legacy aliases: `get_read_ci_service`, `get_search_service`, ? (same providers)

Handlers call `await read_service.method(...)` on native async read services.

Auth JWT / API-key lookup ? `core/auth_lookup.py` (`AsyncUserRepository` on async read pool).

Worker / rate-limit / postgres cache ? `await run_sync_on_write_session` (no thread-pool DB fallback).

## Frontend patterns

- **Page hooks**: `useXPage` holds state + queries; page component is thin
- **Mutations**: prefer `useApiMutation` + `shared/queryInvalidation.ts`
- **Query keys**: centralized in `shared/queryKeys.ts`
- **Forms**: React Hook Form + Zod schemas in `lib/forms/schemas/`; shared `FormField` wrapper
- **Cross-feature modals**: reusable flows (e.g. autodiscover, import type mapping) live in `shared/components/`, not under `features/`
- **Tables**: `VirtualDataTable` (`@tanstack/react-virtual`) + server-side pagination for large lists
- **Modals**: canonical `Modal` / `AppDialog` in `shared/components/ui-primitives.tsx`, exported via `components/ui.tsx`

## Async data access (API)

`DATABASE_ASYNC_ENABLED=true` by default (PostgreSQL via asyncpg).

| Path | Import | When |
|------|--------|------|
| **HTTP reads (v1)** | `Depends(get_*_read_port)` | Native `Async*ReadService` or `SyncReadPort` fallback |
| **HTTP writes (v1)** | `Depends(get_*_write_port)` | Async write pool: native ORM or ``session.run_sync`` bridge; transactional routes excepted |
| **Writes (legacy)** | `core.db_access.run_write` | Worker / internal callers only |
| **Transactional writes** | `run_in_threadpool` / sync `Session` | Routes with `get_transactional_db` (deferred commit at request end) |
| **Fallback** | `core.db_access.run_sync` | `DATABASE_ASYNC_ENABLED=false` |

`core/db_access.py` wraps `async_db.run_db` with named helpers (`run_read`, `run_write`, `run_sync`) and a decision-tree docstring ? prefer it over calling `run_db` directly in new routes.

Optional `DATABASE_READ_URL` routes async reads to a replica.

### Choosing a DB access path

| Need | Use |
|------|-----|
| Default HTTP read (v1) | Inject `get_*_read_port`; `await service.method(...)` |
| Default HTTP write / CRUD | Inject `get_*_write_port`; `await service.method(...)` |
| Multi-step import / autodiscover apply | `get_transactional_*` + deferred commit at request end |
| Unit tests / `DATABASE_ASYNC_ENABLED=false` | `await run_sync(fn)` + sync `Session` |
| Worker / scheduler | `run_worker_write()` |

Pool sizing (`DATABASE_POOL_SIZE`, `DATABASE_MAX_OVERFLOW`, default 3+5): each uvicorn worker holds **read + write** async pools. Safe upper bound:

`api_replicas ? UVICORN_WORKERS ? (pool_size + max_overflow) ? 2 < postgres_max_connections`

With k8s defaults (HPA max 4, workers 2): ceiling ? **128** connections. Use PgBouncer before raising HPA or pool caps.

### Async/sync read consolidation (complete for HTTP read paths)

Sync services and `Async*ReadService` adapters share response assembly in **`services/domain/`** for all domains with async read twins (CI, audit, dashboard, relations, search, correlation, users, ci_types, graph_layout, autodiscover, import/export reads, topology guards). Graph/topology **algorithms** (BFS, impact, business path, chain check) live in **`services/rsm/topology_algorithms.py`** with thin sync/async repo adapters in `graph` / `async_graph` and `topology` / `async_topology`.

### Read path mental model (post-consolidation)

| Layer | Role |
|-------|------|
| `core/deps/read.py` | Request-scoped `Async*ReadService` (or `SyncReadPort` fallback) |

## Background work & scaling

| Process | Command | Responsibility | DB access |
|---------|---------|----------------|-----------|
| API | `uvicorn src.main:app` | HTTP; `BACKGROUND_TASKS_ENABLED=false` in multi-replica prod | Async read/write pools |
| Worker | `python -m src.worker` | Auto-sync scheduler + webhook outbox retry | `run_worker_write()` |

Worker batches call `core/worker_db.run_worker_write()` ? sync ORM via `AsyncSession.run_sync` on asyncpg when `DATABASE_ASYNC_ENABLED=true`.

| Setting | Purpose |
|---------|---------|
| `WEBHOOK_SYNC_DELIVERY=false` | Enqueue only; worker delivers with retries |
| `REDIS_URL` | Distributed cache (`CACHE_ENABLED=true` in k8s when Redis is deployed) |
| `DATABASE_READ_URL` | Read replica for dashboard/graph/search (optional; set in secrets) |
| `DATABASE_POOL_SIZE` / `DATABASE_MAX_OVERFLOW` | Per-engine connection caps (default 3/5) |

K8s manifests: `k8s/` (API HPA, web HPA, worker, web, migration Job). HA Postgres sample: `docker/docker-compose.ha.yml`.

**Worker scaling:** keep scheduler at 1 replica (advisory lock). Outbox consumer can scale horizontally (`SKIP LOCKED`). Split into separate deployments when auto-sync load grows.

**Write scaling:** imports/autodiscover/correlation write to the primary; read replicas do not offload writes ? queue heavy imports or shard by tenant if write load dominates.
