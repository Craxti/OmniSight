# Шпаргалка — ответы по OmniSight РСМ (на демо-данных)

**Русская версия** · [English](PRESENTATION.md)

## 1. Зачем система, какие задачи решает, чем лучше других

### Зачем OmniSight РСМ

В мониторинг одновременно шлют **разные системы** — Zabbix, Prometheus, CMDB, APM. У каждой свой формат: hostname, IP, externalId, serviceCode. Без **единой карты зависимостей** дежурный и correlation engine (CE) вручную выясняют: это один сбой или четыре, какой бизнес-сервис пострадал, с чего начинать расследование.

OmniSight — платформа для учёта ИТ-ландшафта. Модуль РСМ (ресурсно-сервисная модель) ведёт каталог конфигурационных единиц (CI), направленных зависимостей между ними, визуализацию топологии и API для внешних систем мониторинга и корреляции алертов.

**OmniSight РСМ** (ресурсно-сервисная модель) — **источник правды** о том, как устроен ИТ-ландшафт: каталог конфигурационных единиц (CI) — серверы, БД, приложения, бизнес-сервисы — и **направленные связи** между ними (`depends_on`, `hosted_on`, `part_of` и др.). У каждого CI — внешние идентификаторы, по которым алерты из разных систем находят **один и тот же узел**.

**В одной фразе для зала:** не мониторинг и не «мозг» корреляции — **живая карта**, на которую CE и эксплуатация опираются при инциденте.

---

### Какие проблемы решает (боль на инциденте)

| Проблема без РСМ | Что даёт OmniSight |
|---|---|
| Четыре алерта за минуту — **один инцидент или четыре?** | Проверка: все CI на **одной цепочке `depends_on`** → сигнал CE `chain_related` |
| Алерт с hostname и алерт с IP — **про один узел?** | Resolve по любому внешнему ID → один CI в модели |
| Упала БД — **какой бизнес-сервис** затронут? | Граф и impact: путь БД → приложение → … → бизнес-сервис |
| Дежурный **вручную** ищет связи в Excel, чатах, головах коллег | Готовая топология и зона первопричины на экране |
| CMDB **устарела** (сменился IP — модель врёт) | Autodiscover: данные с сервера, человек подтверждает |
| Непонятно, **кто менял** модель и можно ли ей доверять | Аудит с diff по каждому изменению |

---

### Какие задачи закрывает (по ролям)

| Кто | Задача | Как РСМ помогает |
|---|---|---|
| **Дежурный L2/L3** | Быстро понять масштаб и первопричину | Корреляция: сопоставление алертов, зона первопричины, граф цепочки |
| **Correlation Engine** | Решить: склеивать тикеты или нет | API ingest: `chain_related`, enrichment, граф — **готовый контекст** одним вызовом |
| **Инженер мониторинга / SRE** | Связать алерты разных систем с одним объектом | Единый реестр внешних ID (hostname, IP, serviceCode…) |
| **Архитектор / аналитик** | Поддерживать актуальную модель контура | Реестр CI, связи, импорт/экспорт, Autodiscover |
| **IT-директор / владелец сервиса** | Ответ «что затронуто по бизнесу» | Карта: business path, impact от выбранного узла |

---

### Чем OmniSight **не** является (границы — важно сказать на демо)

| Система | Что делает | Чем отличается от РСМ |
|---|---|---|
| **Zabbix / Prometheus / Nagios** | Собирают метрики, шлют алерты | РСМ **не мониторит** — хранит **модель** и **связи**, а не поток метрик |
| **Correlation Engine** | Группирует события, создаёт INC, применяет правила | РСМ **не склеивает тикеты** — отдаёт флаг `chain_related` и данные; решение за CE |
| **Классическая CMDB / Excel** | Справочник активов | РСМ заточен под **корреляцию и граф зависимостей**, а не просто учёт железа |

---

### Чем лучше «других» (сравнение для аудитории)

**Против мониторинга** — мониторинг говорит *«что-то красное»*, РСМ говорит *«это demo-db, от него зависит PAY, начни расследование здесь»*.

**Против CE без модели** — CE видит только поля алертов. Без РСМ он **угадывает** связи; с РСМ получает **эталонную цепочку** `depends_on` и не склеивает ложно (сервер + приложение в одном контуре — разные инциденты, см. сценарий B).

**Против Excel / разрозненной CMDB:**

| Excel / устаревшая CMDB | OmniSight РСМ |
|---|---|
| Всё вносится и правится руками | **Корреляция** — пакет алертов сразу проверяется по готовой карте |
| Карта отстаёт от реальности (IP сменился — алерт «в никуда») | **Autodiscover** — IP и связи с сервера, человек подтверждает |
| CE и дежурный сами сопоставляют поля в разных системах | **Одна актуальная модель** для всех через **API** |
| Нет единого графа для impact и первопричины | **Карта** с business path и подсветкой цепочки |
| Изменения непрозрачны | **Аудит** — кто, когда и что поменял |

**Против «нарисуем в Visio»** — Visio не resolve-ит алерты, не отдаёт JSON в CE и не обновляется с сервера.

---

### Три вопроса, на которые РСМ отвечает на демо

1. Алерт с hostname и алерт с IP — **про один узел**?
2. Упала БД — **какой бизнес-сервис** пострадает?
3. Четыре алерта за минуту — **один инцидент** или четыре?

**Главная ценность:** не красивый UI, а **уверенность на инциденте** — быстрее понять «один сбой или несколько», куда эскалировать, что затронуто по бизнесу.

## 3. Корреляция — пошагово: что нажимать, что на экране, что говорить

**Путь:** меню слева → **Корреляция** (`/correlation`).

**Зачем эта вкладка:** сюда попадают «сырые» идентификаторы из Zabbix, Prometheus, CMDB, APM — как пришли, без ручного сопоставления. РСМ **находит CI в модели**, строит **контекст** (граф, обогащение, зону первопричины) и отвечает CE: алерты лежат на **одной цепочке `depends_on`** или это **разные инциденты**.

> OmniSight **не** склеивает тикеты сам — он отдаёт флаг `chain_related` и данные. Решение «один INC или два» принимает **Correlation Engine** по своим правилам.

---

### 3.0. Экран до нажатия «Обработать»

| Зона экрана | Что видим | Что сказать аудитории |
|---|---|---|
| Заголовок | «Корреляция алертов» | «Имитируем пакет алертов из мониторинга — как CE получает их по API» |
| Подсказка «С чего начать» | 3 пункта (ввод → Обработать → результат) | Можно свернуть после первого показа |
| Таблица **Идентификаторы алертов** | Строки с полями: `hostname`, `ip`, `serviceCode`, `applicationCode`, `externalId`, `cmdbId` | «Разные системы шлют **разные поля** — в модели у CI записаны все внешние ID» |
| Кнопка **Строка** | Добавляет пустую строку | Для сценария A нужно 4 строки |
| Кнопка **Обработать** | Запускает Ingest (вызов API) | «Одним кликом — resolve + проверка цепочки + контекст» |
| Корзина в строке | Удаляет строку | Перед сценарием B — очистить ввод или удалить лишние строки |

**Важно при вводе:** в каждой строке для демо заполняем **только нужные** поля, остальные оставляем пустыми. Пустые поля не мешают — система ищет по тому, что пришло.

### 3.1. Что появляется после «Обработать» (разбор блоков)

После обработки страница заполняется снизу. Идти **сверху вниз** — так удобнее вести демо.

#### Блок «Результат» (слева)

| Элемент | Где | Значение (сценарий A) | Значение (сценарий B) | Что это значит |
|---|---|---|---|---|
| Бейдж | Крупная плашка под заголовком | **Один инцидент** (зелёный, галочка) | **Разные инциденты** (жёлтый, разветвление) | Главный сигнал для CE: `chain_related: true` / `false` |
| **Сопоставлено** | Строка счётчика | **4** | **2** | Сколько алертов нашли CI в модели |
| **Не сопоставлено** | Строка счётчика | **0** | **0** | Алерты «в никуда» — модель не знает такой ID |
| **Неоднозначное сопоставление** | Если > 0, жёлтым | 0 | 0 | Один ID попал на несколько CI — нужна ручная проверка |
| Жёлтое предупреждение | Редко на демо | — | — | Кэш resolve устарел — перезапуск API или повтор через ~30 с |

**Фраза для зала:** «Зелёный бейдж — не „красиво“, а **разрешение склеить** четыре тикета в один. Жёлтый — **не склеивать**, даже если оба алерта в контуре PAY».

#### Блок «Обогащение» (справа, только если есть сопоставления)

Карточки найденных CI: имя, тип (Application, Database…), среда, критичность.

**Что сказать:** «CE и дежурный сразу видят **кто это** в модели — не нужно открывать CMDB и искать `10.0.0.5` руками».

#### Блок «Зона первопричины» (на всю ширину, только сценарий A)

| Элемент | Что видим | Что сказать |
|---|---|---|
| Заголовок | «Зона первопричины» | «С чего начинать расследование» |
| Подсказка | Текст про цепочку `depends_on` | «Ближе к **корню** зависимостей — вероятнее первопричина, не симптом наверху» |
| Карточка CI | **`demo-db`**, тип Database | «Три алерта указывали на БД разными полями — зона сходится на **одном узле**» |

В сценарии B этого блока **нет** — общего каскада `depends_on` между двумя алертами нет, указывать «первопричину» было бы вводить в заблуждение.

#### Граф под зоной первопричины (только сценарий A)

| Визуал | Легенда | Смысл |
|---|---|---|
| Узел с меткой **Первопричина** | «Кандидат в первопричину» | `demo-db` — старт расследования |
| Узлы с меткой **Алерт** | «Элемент, сопоставленный с алертом» | `demo-app`, `demo-db`, `demo-biz` и промежуточные CI на пути |
| Подсвеченные рёбра | Связи `depends_on` между ними | Одна цепочка: БД → приложение → … → бизнес-сервис |

**Фраза:** «APM кричит про `PAY`, Zabbix про `app-01` — на графе видно, что это **один каскад**, а не четыре несвязанных сбоя».

#### Блок «JSON» (внизу, на всю ширину)

Полный ответ API для CE / интеграции. Ключевые поля:

```json
"correlation": {
  "chain_related": true,
  "chain_algorithm": "depends_on_directed",
  "resource_ids": [ ... ],
  "potential_root_cause_zone": [ { "id": "demo-db", ... } ],
  "enrichment": [ ... ],
  "graph": { "nodes": [...], "edges": [...] }
}
```

Кнопка **JSON** (рядом с «Обработать») — скачать тот же payload файлом.

**Фраза:** «CE не рисует граф сам — он забирает **готовый контекст** одним вызовом `/correlation/ingest`».

### Сценарий A — объединить (один инцидент)

| | |
|---|---|
| **Тип** | **Одна цепочка `depends_on`.** Разные системы шлют разные поля, но все алерты попадают на узлы одного каскада: БД → приложение → … → бизнес-сервис PAY. |
| **Зачем показываем** | Ответ на вопрос дежурного: *«Четыре алерта за минуту — один сбой или четыре?»* Без РСМ CE и оператор сами сопоставляют hostname, IP, externalId и коды сервиса. |
| **Что даст** | **Correlation engine** получает сигнал: алерты **можно** склеить в **один INC**. Оператор сразу видит **зону первопричины** (ближе к БД) и затронутый бизнес — не разбирает четыре несвязанных тикета. |

**Легенда для выступления:** 02:00, платёжный контур. Zabbix, Prometheus, CMDB и APM одновременно шлют тревоги — поля разные, смысл один.

#### Шаги на экране

| Шаг | Действие | Что видим | Что говорим |
|:---:|---|---|---|
| 1 | Открыть `/correlation` | Пустая таблица, одна строка | «Сейчас прилетело 4 алерта из разных систем» |
| 2 | **Строка** ×3 — всего **4 строки** | 4 пустые строки | — |
| 3 | Заполнить **только** указанные ячейки (остальное пусто) | См. таблицу ниже | «У каждой системы свой формат — РСМ знает все ID в модели» |
| 4 | **Обработать** | Скелетон загрузки, затем результат | «Resolve + проверка цепочки — один запрос» |
| 5 | Блок **Результат** | Бейдж **Один инцидент**, Сопоставлено **4**, Не сопоставлено **0** | «Все четыре нашли CI — это уже победа над Excel» |
| 6 | Блок **Обогащение** | Карточки `demo-app`, `demo-db`, `demo-biz`… | «Тип, критичность — без перехода в реестр» |
| 7 | **Зона первопричины** | Карточка **`demo-db`** | «Начинаем с БД — три алерта сходятся сюда разными полями» |
| 8 | **Граф** | Цепочка с метками Первопричина / Алерт | «Визуально — один каскад, не четыре острова» |
| 9 | **JSON** | `chain_related: true`, `potential_root_cause_zone` | «Так CE получает ответ для правила „один инцидент“» |

**Ввод** — 4 строки, в каждой **одно** (или пара для APM) заполненное поле:

| Строка | Как в мониторинге | Поле в таблице | Значение | Куда попадает в модели |
|:---:|---|---|---|---|
| 1 | Zabbix | `hostname` | `app-01` | **demo-app** (Application) |
| 2 | Prometheus | `ip` | `10.0.0.5` | **demo-db** (Database) |
| 3 | CMDB | `externalId` | `ext-db-1` | **demo-db** (тот же узел, что строка 2) |
| 4 | APM | `serviceCode` + `applicationCode` | `PAY` + `PAY-APP` | **demo-biz** (Business Service) |

**Логика для себя (не обязательно вслух):** все четыре CI лежат на одном пути `depends_on`:

```
demo-db → demo-app → demo-pay-it → demo-biz
```

Строки 2 и 3 — **дубликат по смыслу** (одна БД, два внешних ID) — показываем, что resolve работает с любым полем.

**Итог для CE:** `chain_related: true` → правило «один инцидент». Первопричина — у БД (`demo-db`), не у APM-кода `PAY` наверху цепочки.

### Сценарий B — не объединять (разные инциденты)

| | |
|---|---|
| **Тип** | **Два узла, одна цепочка `depends_on` — нет.** Оба алерта в платёжном контуре и оба находятся в модели, но лежат на **разных ветках** графа (сервер по `hosted_on`, приложение по `depends_on`). |
| **Зачем показываем** | Ответ на вопрос: *«Два алерта в одном контуре — всегда один инцидент?»* Нет. Совпадение по времени и домену PAY **не** означает общую первопричину. |
| **Что даст** | CE **не** получает основание склеивать: **два отдельных INC** (диск на сервере и ошибка приложения — разные работы). РСМ **не даёт ложного «да»** на объединение инфраструктуры и приложения. |

**Легенда для выступления:** в том же PAY упали и хост `pay-srv-01`, и приложение `app-01`. Время близкое, контур один — но это **два разных узла** без связи `depends_on` между ними.

| Шаг | Действие | Что видим | Что говорим |

| 1 | **Очистить** ввод после сценария A (удалить строки 3–4 или очистить все поля) | 1–2 пустые строки | «Другая ночь — два алерта, оба с hostname» |
| 2 | **2 строки**, в каждой только `hostname` | См. таблицу ниже | «Формат одинаковый — но узлы в модели разные» |
| 3 | **Обработать** | Результат без зоны первопричины | — |
| 4 | Блок **Результат** | Бейдж **Разные инциденты** (жёлтый), Сопоставлено **2** | «Оба **сопоставились** — модель не сломана» |
| 5 | Нет блока **Зона первопричины** | Пусто под обогащением | «Общего каскада нет — **не угадываем** первопричину» |
| 6 | Нет **графа цепочки** | Только JSON и обогащение (если есть) | «CE видит `chain_related: false` — **два тикета**» |
| 7 | **JSON** | `"chain_related": false` | «Ложного объединения нет — это фича, не баг» |

**Ввод** — 2 строки, только `hostname`:

| Строка | `hostname` | Найденный CI | Тип CI | Почему не склеивается |
|:---:|---|---|---|---|
| 1 | `pay-srv-01` | **demo-pay-srv** | Server | Связан с приложением через `hosted_on`, не `depends_on` |
| 2 | `app-01` | **demo-app** | Application | Идёт по цепочке `depends_on` → БД, **не** к серверу напрямую |

**Схема для себя:**

```
demo-pay-srv  ←hosted_on—  demo-app  —depends_on→  demo-db  —…→  demo-biz
     ↑                           ↑
  алерт 1                    алерт 2
```

Между `demo-pay-srv` и `demo-app` нет ребра `depends_on` в нужном направлении для «одной цепочки» — алгоритм `depends_on_directed` возвращает **false**.

**Итог для CE:** `chain_related: false` → правило «не объединять» (или эскалация оператору). Сопоставление сработало, но **склейки нет** — и это ожидаемое поведение, не ошибка модели.

## 4. Сценарий Autodiscover — что и зачем

**Проблема:** у `demo-app` в реестре есть hostname и коды (`PAY`, `PAY-APP`), но **нет IP**. Алерт только по IP приложения **не сопоставится**.

**Зачем Autodiscover:**

| Роль | Что получает |
|---|---|
| Аналитик | Не дописывать IP в Excel — **скан с сервера** |
| Оператор | Алерты по IP приложения начнут находиться в модели |
| Директор / CE | Корреляция на **актуальной** карте |

**Шаги на демо:**

1. `/inventory` → `demo-app` — **нет IP**
2. **Autodiscover** → галочка **`demo-pay-srv`** → **Ручной режим** → **Сканировать**
3. Вкладка **Поля:** `demo-app` / `ip` / **`10.0.0.10`**
4. Вкладка **Связи:** `depends_on` → `demo-db` (если предложено)
5. **Применить выбранное** → IP появился у `demo-app`
6. `/audit` — запись с diff

**Итог:** модель **не рисовали руками** — подтянули с сервера, человек подтвердил. Это главное отличие от Excel-CMDB: карта **не отстаёт** от реальности, и сценарии корреляции A/B работают надёжнее.

---

**Кратко для выступления:** карта уже загружена (11 CI, 10 связей) → вкладки → **Корреляция:** таблица ID → **Обработать** → бейдж «один / разные инциденты» → обогащение → (A) зона `demo-db` + граф → JSON для CE → **A:** 4 алерта, `chain_related: true` → **B:** сервер + приложение, `chain_related: false` → **Autodiscover:** IP с сервера без ручного CMDB.

**Опционально:** Demo CE (http://localhost:8090) — те же алерты с карточками Zabbix/Prometheus; в UI — **Настройки → API** для ключа и Swagger.
